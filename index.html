    (Topic of the month : Locks and Deadlocks )

    1. Locks and Deadlocks -Locking and Deadlocks are the most reported issue by the Application team. Most of the people use deadlock as a common word for things related to locking. Lock timeout and deadlock are totally different but one common thing , Both a deadlock and a lock timeout return an SQL0911 error code. A deadlock is return RC-2, while a lock timeout is return RC-68.
     
    2. Lock-Timeout and Deadlocks -Even when you try to read the data from the table, It will cause locking until and unless you are using proper isolation level. When some other connection is holding on to a lock that is not compatible with the lock you want, you must wait and application status change to lock-wait. Lock-waiting causes drop in response time. Application will in lock-wait state till the value set in LOCKTIMEOUT parameter at DB level. Once time period got over , Waiting application will fail with locktimeout (SQL911N , RC-68) By Default LOCKTIMEOUT value is -1 i.e infinite time. We need to change it as per the need. For OLTP it is recommended set between 30 to 60 seconds and for OLAP it can be little higher. A deadlock occurs when two or more application processes each hold locks on resources that the others need and without which they cannot proceed. After a preset time interval, Db2 can roll back the current unit of work for one of the processes or request a process to terminate. In determining which process to roll back or terminate, Db2 assesses many characteristics of the processes that are involved in the deadlock and chooses the one that, if terminated, will cause the least impact relative to the other processes. By choosing a process to roll back or terminate, Db2 frees the locks and allows the remaining processes to continue. .
     
    2. Tracking and Analysing: -Lock-Wait/Timeout We can always monitor lock-wait application from db2top or dsmtop utility. Through these utilities it is always easy to check the waiting agent and blocker agent. Also we get the details due to which object this locking is occuring. Incase above utilities are not working then we can collect the db snapshot to check these details. Deadlocks: It pretty difficult to monitor the deadlock occurance untill and unless some monitoring is placed to monitor the connections, So we can enable event monitor. By default deadlock monitor is enabled on Db2 DB. We can check Event monitor details by accessing data from SYSIBM.SYSEVENTMONITORS tables. Since it is difficult to login and monitor the deadlock , We can always enable the event monitor to keep the track of it. Its very helpful in analysing the data to find the root cause of locking or deadlock. There are simple steps which we need to follow. Please check below link for more details on event monitoring: https://www.ibm.com/support/knowledgecenter/SSEPGG_11.1.0/com.ibm.db2.luw.admin.mon.doc/doc/c0059011.html
