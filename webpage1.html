<p>(Topic of the month : Performance tuning )</p>

<p><div style="text-align: justify;"><span style="color:#000000"><span style="font-family:times new roman,times,baskerville,georgia,serif"><span style="font-size:18px">1. <a href="https://www.ibm.com/support/knowledgecenter/en/SSEPGG_11.1.0/com.ibm.db2.luw.admin.perf.doc/doc/c0005414.html" target="_blank" style="mso-line-height-rule: exactly;-ms-text-size-adjust: 100%;-webkit-text-size-adjust: 100%;color: #007C89;font-weight: normal;text-decoration: underline;">Performance tuning</a> -</span><span style="font-size:16px">Performance refers to the way that a computer system behaves in response to a particular workload. Performance is measured in terms of system response time, throughput, and resource utilization.

.Performance is also affected by: 1) The resources that are available on the system 2) How well those resources are used and shared </span></span></span><br>

&nbsp;


<div style="text-align: justify;"><span style="color:#000000"><span style="font-family:times new roman,times,baskerville,georgia,serif"><span style="font-size:18px">2. <a href="https://www.ibm.com/support/knowledgecenter/SSEPGG_11.1.0/com.ibm.db2.luw.admin.perf.doc/doc/t0070377.html" target="_blank" style="mso-line-height-rule: exactly;-ms-text-size-adjust: 100%;-webkit-text-size-adjust: 100%;color: #007C89;font-weight: normal;text-decoration: underline;">Baseline database performance</a> -</span><span style="font-size:16px"> Baseline database performance: You need to know what is the normal performance measure of you database so that when you observer some complaint from application uses you can check your database performance and compare with baseline statiscics to arrive if it is indeed database which is impacting application performance.

Starting Db2 11.1 Db2 ships db2mon script under ~/sqllib/samples/perf directory. You can use db2mon script to benchmark your database performance under specific/normal scenario. .</span></span></span><br>

&nbsp; 



<div style="text-align: justify;"><span style="color:#000000"><span style="font-family:times new roman,times,baskerville,georgia,serif"><span style="font-size:18px">2. <a href="https://www.ibm.com/support/pages/db2mon-script-monitoring-performance" target="_blank" style="mso-line-height-rule: exactly;-ms-text-size-adjust: 100%;-webkit-text-size-adjust: 100%;color: #007C89;font-weight: normal;text-decoration: underline;">db2mon script for monitoring performance</a> -</span><span style="font-size:16px"> If you make any changes in the system for ex. any hardware change, application change, new application deployment, database change etc. you can use db2mon to measure the database performance before and after the change. Later you can compare the two db2mon reports to know how good/bad the change is for your database.


In the db2mon report file, you can check for - Throughput metrics at database level

Performance troubleshooting is a challenging topic. This script is widely used within support to troubleshoot performance problems, it is generic in nature and works well with any flavour of DB2 ( including Purescale )

Instead of traditional snapshot monitoring, this uses the in-memory metrics interfaces available for obtaining various performance details about the database.Tip: For costly SQLs see -Top SQL statements by execution time.Tip:

The "Statement and Plan Identifiers" section shows the EXECUTABLE_ID for each of the top 100 statements. You can use the EXECUTABLE_ID with the MON_GET_PKG_CACHE_STMT function to get the access plan for any of these statements. You can also use EXECUTABLE_ID to determine the explain plan, by using the following command:

db2 "call explain_from_section(x'<executable id>','M',NULL,0,'<current user>',?,?,?,?,?)" </span></span></span><br>
